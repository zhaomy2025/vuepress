# Java要彻底告别32位系统啦！这对你的开发有什么影响？

关于Java弃用并移除32位x86端口，这确实是近年来Java平台一个重要的方向性调整。简单来说，这个过程是**分阶段进行的**：从**Java 21开始弃用**，并在**Java 25中最终完成移除**。

## 🗓️ 移除时间线与关键版本

下面的表格清晰地展示了这一政策在各个Java版本中的推进过程：

| JDK 版本 | 状态 | 相关 JEP | 说明 |
| :--- | :--- | :--- | :--- |
| **JDK 21** | **弃用 (Windows)** | JEP 449: 弃用 Windows 32 位 x86 端口 | 标志着官方开始淘汰32位支持 |
| **JDK 24** | **删除 (Windows)** | JEP 479: 删除 Windows 32 位 x86 端口 | 移除了对Windows 32位x86系统的支持 |
| **JDK 24** | **弃用 (Linux)** | JEP 501: 弃用 32 位 x86 端口 | 弃用对Linux 32位x86系统的支持 |
| **JDK 25 (LTS)** | **删除 (Linux)** | JEP 503: 移除 32 位 x86 端口 | 移除了对Linux 32位x86系统的支持 |

### JEP 449: 弃用 Windows 32 位 x86 端口

::: info JEP 449: Deprecate the Windows 32-bit x86 Port for Removal
Deprecate the Windows 32-bit x86 port, with the intent to remove it in a future release.
:::

弃用 Windows 32 位 x86 端口，并计划在未来版本中将其移除。

### JEP 479: 删除 Windows 32 位 x86 端口

::: info JEP 479: Remove the Windows 32-bit x86 Port
Remove the source code and build support for the Windows 32-bit x86 port. This port was deprecated for removal in JDK 21 with the express intent to remove it in a future release.
:::

移除针对 Windows 32 位 x86 端口的源代码和构建支持。该端口已在 JDK 21 中被标记为弃用并计划移除，并明确表达了在未来的版本中将其移除的意图。

### JEP 501: 弃用 32 位 x86 端口并将其删除

::: info JEP 501: Deprecate the 32-bit x86 Port for Removal
Deprecate the 32-bit x86 port, with the intent to remove it in a future release. This will thereby deprecate the Linux 32-bit x86 port, which is the only 32-bit x86 port remaining in the JDK. It will also, effectively, deprecate any remaining downstream 32-bit x86 ports. After the 32-bit x86 port is removed, the architecture-agnostic Zero port will be the only way to run Java programs on 32-bit x86 processors.
:::

弃用 32 位 x86 端口，并计划在未来版本中将其移除。此举将导致目前 JDK 中唯一剩余的 32 位 x86 端口——Linux 32 位 x86 端口被弃用，同时也将实质上弃用所有下游衍生的 32 位 x86 移植版本。在该端口被移除后，与架构无关的 Zero 移植版本将成为在 32 位 x86 处理器上运行 Java 程序的唯一方式。

### JEP 503：移除 32 位 x86 端口

::: info JEP 503: Remove the 32-bit x86 Port
Remove the source code and build support for the 32-bit x86 port. This port was deprecated for removal in JDK 24 (JEP 501) with the express intent to remove it in a future release.
:::

移除32位x86端口的源代码和构建支持。该端口在JDK 24（JEP 501）中已被标记为弃用，明确计划在未来版本中移除。

作为这一过程的最终步骤，**JEP 503** 的核心内容如下：

- **主要内容**：在JDK 25中，**移除了对Linux 32位x86端口的支持**。这与之前在JDK 24中移除Windows 32位支持一脉相承，标志着32位x86架构在主流Java生态系统中的谢幕。
- **最终影响**：在此之后，**所有在32位x86处理器上运行的Java应用程序都将没有官方的OpenJDK端口可供使用**。

## 🤔 为何要移除32位支持？

这一决策背后有深刻的技术和市场原因：

- **技术债务与创新阻碍**：维护陈旧的32位端口需要持续的开发和测试资源，这拖慢了在新功能（如**虚拟线程**）上的创新步伐。事实上，32位Windows系统由于技术限制，无法充分享受虚拟线程带来的性能优势。
- **操作系统支持生命周期**：主流操作系统已经或正在结束对32位版本的支持。例如，最后一个32位Windows主流版本的支持已于2025年10月结束。继续支持一个已被厂商放弃的平台意义不大。
- **性能考量**：虽然在64位系统上可以通过WOW64层运行32位程序，但这会带来显著的**性能下降**。官方不再提供为该架构优化的JDK，也是情理之中。
- **市场需求减少**：随着硬件价格的下降和64位系统的普及，使用32位系统的用户比例已经极低，继续维护这些端口的投资回报率越来越低。

## 💡 给你的建议

- **升级硬件和系统**：最根本的解决方案是将你的应用环境迁移到**64位操作系统**和**64位JDK**。这是技术发展的必然趋势，也能让你的应用获得最佳性能，包括虚拟线程等现代Java特性带来的优势。
- **暂留旧版本**：如果你的业务确实暂时无法离开32位环境，唯一的办法就是**停留在最终的、支持32位的JDK版本上**（对于Windows 32位，最后一个版本是JDK 23；对于Linux 32位，最后一个版本是JDK 24）。但请注意，这将无法获得后续的安全更新和新功能。
- **提前规划迁移**：如果您计划升级到Java 25（LTS），请务必在此之前完成向64位环境的迁移，以避免兼容性问题。
- **测试兼容性**：在迁移到64位环境前，应全面测试应用程序的兼容性，特别是那些依赖于特定内存模型或平台特性的应用。

## 领域算法

### 安全算法

- 摘要算法：~~MD4（已弃用）~~、~~MD5（已弃用）~~、~~SHA-1（已弃用）~~、SHA-2系列、SHA-3系列
  - SHA-2系列：SHA-224、**SHA-256（应用最广泛）**、SHA-384、SHA-512、SHA-512/224、SHA-512/256
  - SHA-3系列：基于海绵结构的新一代哈希函数，包括 SHA3-224、SHA3-256、SHA3-384、SHA3-512
  - 其他哈希函数：BLAKE2系列、Argon2、Scrypt
    - BLAKE2系列：BLAKE2b、BLAKE2s、BLAKE2bp/BLAKE2sp
    - Argon2：2015年密码哈希竞赛获胜算法，专为密码存储设计
    - Scrypt：内存密集型哈希函数，用于比特币挖矿和密码存储
    - SM3：密码哈希函数，类似于SHA-256
- 对称加密算法：
  - 分组密码：DES、3DES、**AES**
    - DES：第一个面向公众的加密标准，56位密钥
    - 3DES：三重DES，主要用于遗留系统兼容场景
    - AES：最常用的对称加密算法，支持128/192/256位密钥长度
    - SM4：对称加密算法，类似于AES，用于数据加密
  - 流密码：RC4、**ChaCha20**
    - RC4：传统流密码，因安全性问题已不推荐用于新系统
    - ChaCha20：现代流密码，比RC4更安全高效，与Poly1305配合使用
- 消息认证码（MAC）：确保消息完整性、认证发送者身份、防止重放攻击
  - **HMAC**：基于哈希函数的消息认证码，是最广泛使用的MAC算法之一
    - 支持多种哈希算法变体（HMAC-MD5、HMAC-SHA1、HMAC-SHA256等）
    - 应用于SSL/TLS协议、API接口认证等
  - CMAC：基于分组密码的消息认证码，适用于与分组密码结合使用的场景
    - 支持任何128位分组密码（如AES-CMAC）
    - 应用于金融交易系统、智能卡等
  - **Poly1305**：高效的消息认证码算法，通常与ChaCha20配合使用
    - 应用于TLS 1.3协议、高速数据传输系统等
  - GMAC：基于Galois/Counter Mode (GCM)的认证码，常用于认证加密模式
    - 应用于AES-GCM、磁盘加密等
- 非对称加密算法：理论上都可用于数字签名，但常用的是RSA（RS256）和ECC（**ECDSA**、**EdDSA**）
  - **RSA**：基于大整数因子分解，安全性好但密钥较长，可用于加密和数字签名
  - Rabin：基于大整数因子分解，可证明安全性与大整数分解等价。理论上可用于数字签名，但实际不常使用，主要原因是解密（或签名验证）时会产生四个可能的解，需要额外机制选择正确解，增加了复杂性和实现难度
  - ElGamal：基于离散对数，支持概率性加密，密文长度是明文两倍，可用于加密和数字签名
  - **ECC**：基于椭圆曲线，相同安全性下密钥更短，计算效率更高，可用于加密、密钥交换（ECDH）和数字签名（ECDSA、EdDSA）
  - SM2：基于椭圆曲线的非对称加密算法，用于数字签名、密钥交换和公钥加密
- 数字签名算法：RS256、~~DSA（已逐渐淘汰）~~、**ECDSA**、**EdDSA**
  - RS256：基于RSA的数字签名算法，使用SHA-256作为哈希函数，传统领域主流选择
  - DSA：基于离散对数问题，专门用于数字签名
  - ECDSA：基于椭圆曲线离散对数问题，是DSA的椭圆曲线版本，广泛应用于加密货币等领域
    - secp256r1（NIST P-256）
    - secp256k1（比特币用）
  - EdDSA：基于Edwards曲线，是ECDSA的改进版，更简单安全，性能更好
    - Ed25519（基于 Curve25519）
- 国密算法：SM2（非对称加密算法）、SM3（摘要算法）、SM4（对称加密算法）
- 密钥交换协议：**DH**、**ECDH**
  - DH：基于离散对数问题，允许双方在不安全通道协商共享密钥
  - ECDH：基于椭圆曲线，是DH的升级版本，效率更高
  - DHE：DH的临时版本，支持前向安全性，每次会话生成新密钥对
  - ECDHE：ECDH的临时版本，结合椭圆曲线高效性与前向安全性，TLS 1.3强制使用
- 密钥派生函数
  - HKDF：基于HMAC的密钥派生函数，用于从主密钥派生出多个子密钥，支持多种哈希算法
- 加密模式
  - 第一代加密模式：基本加密模式
    - ECB
  - 第二代加密模式：安全增强模式
    - CBC
    - CTR
    - CFB
    - OFB
  - AEAD：同时提供数据机密性、完整性和真实性保护，简单说就是加密＋认证
    - AES-GCM
    - ChaCha20-Poly1305

**非对称加密相关算法对比**

| 基于的数学难题 | 非对称加密 |  数字签名 |密钥交换 |
| :----------- | :------- | :----- | :----- |
| 大整数分解问题 | RSA、Rabin | RSA | - |
| 离散对数问题 | ElGamal |  DSA | DH |
| 椭圆曲线离散对数问题 | ECC | ECDSA、EdDSA | ECDH |

> **注1**：理论上非对称加密都可用于数字签名，但常用的是RSA、ECDSA、EdDSA
> - DSA 已逐渐淘汰
> - Rabin 解密时会产生四个可能的解，需要额外机制选择正确解
> - ElGamal 理论上可构造签名方案，但不如 DSA 高效

> **注2**：关于ECC可用于密钥交换而RSA不常用于此场景的说明：
> - **RSA理论上的密钥交换方式**：RSA可通过"公钥加密-私钥解密"方式交换密钥（一方用另一方公钥加密预共享密钥），但这不是专门的密钥交换协议。
> - **ECC的密钥交换优势**：ECC基于椭圆曲线离散对数问题，天生适合与DH密钥交换协议结合，形成高效的ECDH协议。
> - **技术差异**：
>   - ECDH支持**完美前向保密（PFS）**，即使长期私钥泄露，已交换的会话密钥也不会被破解
>   - RSA直接密钥交换不支持PFS，需要额外机制（如TLS中的临时RSA）
>   - ECDH在相同安全强度下，计算效率和带宽消耗远优于传统DH，更适合现代网络通信
> - **应用现状**：现代TLS协议已广泛采用ECDH（而非RSA密钥交换）作为主要的密钥协商方式，以获得更好的安全性和性能。

**消息认证码（MAC）VS 数字签名**

| 对比维度 | 消息认证码（MAC） | 数字签名 |
|---------|-----------------|---------|
| **密钥类型** | 对称密钥 | 非对称密钥对（私钥签名，公钥验证） |
| **不可否认性** | 不提供（发送者和接收者都拥有密钥） | 提供（只有发送者拥有私钥） |
| **验证方范围** | 只有拥有共享密钥的接收者可以验证 | 任何人拥有发送者公钥都可以验证 |
| **计算效率** | 通常更高效（对称密钥运算更快） | 计算开销较大，特别是签名过程 |
| **主要功能** | 消息完整性、发送者认证、防止重放攻击 | 消息完整性、发送者认证、**不可否认性**、防止重放攻击 |
| **应用场景** | 内部系统通信、API接口认证、数据库完整性校验 | 电子合同、软件发布、数字证书、公开通信 |
| **安全性特点** | 依赖对称密钥的安全性，不适合公开验证 | 依赖非对称密钥的安全性，适合公开验证和第三方审计 |
| **实现复杂度** | 相对简单，易于实现和部署 | 相对复杂，需要密钥管理和证书基础设施 |
| **典型算法** | HMAC、CMAC、Poly1305、GMAC | RSA、ECDSA、EdDSA、SM2 |
